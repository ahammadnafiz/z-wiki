You (Claude) are the maintainer. The user curates sources and asks questions. You execute
ingest, compile, query, and lint. Raw sources are immutable. The wiki is yours
to write and revise. The user rarely edits wiki pages by hand.

## Layers
- `raw/` — source material. Read-only for you. Subfolders: `articles/`, `papers/`,
  `transcripts/`, `assets/` (images, PDFs, binaries referenced by sources).
- `wiki/` — your working output. See structure below.
- `templates/` — reference templates. Copy the relevant template when creating
  a new page; do not deviate from the frontmatter schema.

## Wiki structure
- `wiki/index.md` — **thin** top-level nav (<1K tokens: counts + links to
  type catalogs + tag shards). **Generated** by `scripts/shard_index.py`
  during COMPILE / LINT / post-ingest. Do not hand-edit. At scale, loading
  this file into context on every QUERY would dominate the prompt — that's
  why it stays thin and why retrieval starts at `scripts/wiki_search.py`.
- `wiki/log.md` — append-only changelog. Never rewrite, only append. Each entry
  starts with `## [YYYY-MM-DD HH:MM] <op> | <short-title>`.
- `wiki/sources/` — one summary per raw source. Filename: `{slug}.md` derived
  from the raw filename.
- `wiki/concepts/` — one page per concept. Full page if ≥2 sources mention it;
  stub otherwise.
- `wiki/entities/` — one page per person, organisation, library, project, tool.
  Same threshold.
- `wiki/outputs/` — filed answers to my queries and lint reports. QUERY **always**
  writes here; never directly to `syntheses/`.
- `wiki/syntheses/` — cross-cutting analyses. Created **only** when I explicitly
  ask, or when a stable thread of outputs is promoted (crystallized) into a
  synthesis. Syntheses are promoted outputs, not auto-generated from a single query.
- `wiki/attachments/images/` — charts, diagrams, anything you generate (figures
  extracted from sources land here; see Quality bar).
- `wiki/.meta/` — **generated** sidecar cache. `backlinks.json`,
  `sources.json`, `search-index.tsv`. Written by `scripts/build_meta.py`;
  read by every operation that needs backlink counts or grep substrate.
  See `wiki/.meta/README.md`. Derived data — frontmatter stays
  authoritative; LINT reconciles drift.
- `wiki/indexes/` — **generated** shard catalogs. Per-type `_index.md`
  shards live under each type directory; `wiki/indexes/by-tag/{tag}.md`
  exists for every tag with ≥3 members. Written by
  `scripts/shard_index.py`. See `wiki/indexes/README.md`.
- `wiki/views/` — Obsidian Bases (`.base` files). **Ornamental.** Dynamic
  views over frontmatter (tables, cards, list, map). Must not become primary
  navigation. Syntax: `.claude/skills/obsidian-bases/SKILL.md`.
- `wiki/canvases/` — Obsidian JSON Canvas (`.canvas` files). **Ornamental.**
  Visual workspaces for mind-maps, architectural diagrams, dependency graphs.
  Any knowledge expressed in a canvas must also exist in plain markdown
  somewhere in `wiki/`. Syntax: `.claude/skills/json-canvas/SKILL.md`.

## Scale architecture

The vault is designed to scale from dozens to thousands of sources without
the per-query cost growing with vault size. Three mechanisms do the work:

1. **Sidecar cache (`wiki/.meta/`).** `backlinks.json` stores the inverse
   wikilink graph; `sources.json` is the flat source registry;
   `search-index.tsv` is the grep substrate (one line per page: `path \t
   title \t tags \t concepts \t summary`). Regenerated by
   `scripts/build_meta.py`. Every operation that previously had to scan
   the filesystem now reads this cache — O(1) reads replace O(N²) scans.
2. **Sharded index.** `wiki/index.md` stays under ~1K tokens. Per-type
   catalogs live at `wiki/<type>/_index.md` and auto-shard into
   alphabetical slices when they exceed 10K tokens. Tag shards at
   `wiki/indexes/by-tag/{tag}.md`. `scripts/shard_index.py` writes all of
   them atomically from the sidecar.
3. **Hybrid-first retrieval.** `scripts/wiki_search.py` auto-selects:
   - **Lexical** (BM25-style over `search-index.tsv`) when no
     embeddings exist. Always available. Zero deps.
   - **Hybrid** (lexical + local semantic, merged via Reciprocal Rank
     Fusion with k=60) when `wiki/.meta/embeddings.npy` exists. Better
     recall for queries whose keywords don't match the sources'
     vocabulary. Opt in with `/wiki-enable-semantic` — that one
     command installs the deps, computes the first embedding pass,
     and flips everything into hybrid mode. All subsequent
     operations auto-maintain embeddings via `--embed-if-enabled`;
     no further user action needed. To disable, delete
     `wiki/.meta/embeddings.npy` and
     `wiki/.meta/embeddings-manifest.tsv`.
   Both modes apply type-diversity (no single type >60% of top N) and
   Jaccard near-dup dedup. Query cost is bounded by the small ranked
   page set, not by vault size.

Read `.claude/skills/context-engineering/SKILL.md` for how these combine
with Claude context-window management (subagents, rewind, compact/clear).

## Tooling

Core tooling is plain files + Python stdlib + ripgrep. Zero
third-party deps for the core path. Semantic search is opt-in and
adds `numpy` + `sentence-transformers` (see `requirements.txt`).

- `scripts/build_meta.py` — rebuild `wiki/.meta/*`. Idempotent.
  `--check` reports drift without writing. `--embed` additionally
  computes local embeddings (incremental — only pages whose text
  SHA changed).
- `scripts/shard_index.py` — rebuild `wiki/index.md`, per-type
  `_index.md`, and tag shards from the sidecar.
- `scripts/wiki_search.py "<query>"` — hybrid-first ranked search.
  `--mode auto|lexical|semantic|hybrid` (auto = hybrid if
  embeddings exist else lexical); `--type`, `--tag`, `--limit`,
  `--json` flags. Applies type-diversity cap (60%) and Jaccard
  near-dup dedup to results.
- `scripts/semantic.py` — cosine-similarity over stored embeddings.
  Invoked by `wiki_search.py`; usable standalone for debugging.
  Requires sentence-transformers.
- `scripts/check_invariants.sh` — run every test in `tests/`.
  Exits non-zero on any failure.
- `tests/*.sh` — invariant suite (9 tests): wikilinks resolve,
  filenames kebab-case, frontmatter valid, `source_count` matches
  sidecar, `wiki/index.md` stays under the 10K-token cap,
  embeddings manifest matches the `.npy` shape + backlinks graph,
  etc. See `tests/README.md`.

## File conventions
- Filenames: lowercase, kebab-case, ASCII only. No spaces, no underscores, no
  uppercase. `active-inference.md`, not `Active_Inference.md`.
- Source summaries: `{author-lastname}-{year}-{short-title}.md` when author/year
  are known; otherwise `{publisher}-{short-title}.md`.
- All files must have YAML frontmatter matching the schema below.
- Links: `[[shortest-path]]` style. Obsidian resolves by filename because
  filenames are globally unique in this vault — enforce that invariant when
  creating pages. If a name would collide, disambiguate with a suffix
  (`uniswap-protocol.md` vs `uniswap-labs.md`).
- Bold key terms on first use per article. Link first occurrence per section.

## Frontmatter schema (all wiki pages)
```yaml
---
title: "Page Title"
type: source | concept | entity | synthesis | output | stub
status: draft | review | final
date_created: YYYY-MM-DD
date_modified: YYYY-MM-DD
last_seen: YYYY-MM-DD
summary: "One to two sentences. No line breaks."
tags:
  - domain-tag
  - topic-tag
---
```
- `last_seen` — the date any operation (INGEST, QUERY, LINT) last
  substantively read, cited, or wrote this page. Distinct from `date_modified`,
  which tracks body edits only. Used by LINT to surface stale content.

Additional fields by type:
- `source`: `source_url`, `source_path` (relative to vault root), `authors`, `published`, `source_type`. Papers add `venue`, `doi`, `arxiv`, `page_count`.
- `concept` / `entity`: `related` (list of `[[wikilinks]]`), `source_count` (int, **derived** — the number of distinct `wiki/sources/*.md` files that wikilink to this page. Canonically recomputed by COMPILE and LINT; never hand-edited).
- `output`: `question`, `answered_on` (YYYY-MM-DD), `sources_cited`.
- `stub`: `promote_when` — short note on what would justify promoting to full page.

## Operations

### INGEST
Trigger: I say "ingest" or run `/wiki-ingest`, or you see new files in `raw/`
that have no matching summary in `wiki/sources/`.

Batch-size policy (context engineering — this is the single biggest cost
lever):
- **1–2 sources:** ingest inline in the main session.
- **3–10 sources:** dispatch one subagent per source. Each subagent reads
  its raw file in isolation, writes the source summary + any stubs, and
  returns a structured report. Parent session never sees the raw bodies.
- **11+ sources:** dispatch in waves of 10; rebuild `wiki/.meta/` between
  waves so the next wave sees what the previous wave created.
- **Any source > 50K tokens of extracted markdown:** always goes to its
  own subagent, regardless of batch size.

Read `.claude/skills/context-engineering/SKILL.md` for the rationale
and the subagent report format.

Steps:
1. Find unprocessed sources: read `wiki/.meta/sources.json` to get the set
   of processed `source_path:` values; compare against `raw/**/*.{md,txt,
   pdf,docx,pptx,xlsx,xls,html,htm,epub,rtf}`. Any raw file not in the
   set is unprocessed. If `sources.json` is missing or older than the
   newest file in `wiki/sources/`, rebuild it first:
   `python3 scripts/build_meta.py`.
2. For each one:
   a. Read fully.
      - Markdown / text (`.md`, `.txt`): read directly.
      - Binary documents (`.pdf`, `.docx`, `.pptx`, `.xlsx`, `.xls`,
        `.html`, `.htm`, `.epub`, `.rtf`): extract to markdown with
        `markitdown`, not native file input. Run
        `markitdown <path> -o <tmp>.md` and read the emitted markdown.
        `markitdown` preserves headings, lists, tables, and links in a
        token-efficient form that LLMs parse cleanly, and it unifies
        extraction across all document types. Check `markitdown` is on PATH
        first; if it isn't installed, install with
        `pip install 'markitdown[all]'` (or scoped extras like
        `[pdf, docx, pptx, xlsx]` for a smaller footprint). If the output
        is empty or gibberish (e.g. scanned / image-only PDF), fall back to
        rasterizing pages and reading visually.
      - **Token gating** (applies to the extracted markdown for any binary
        document; also applies to `.md` / `.txt` sources read directly):
        - **≤50k tokens** — read the full extraction in one pass.
        - **50k–200k tokens** — recursive chunking. Protocol:
          1. Split the extracted markdown into ~20k-token chunks with
             ~2k-token overlap, respecting heading/section boundaries
             where possible (split on `#`/`##` lines, not mid-paragraph).
          2. Maintain a **running outline** capped at **4K tokens**.
             Content: section headings seen so far + every concept/entity
             slug mentioned + 3–5 load-bearing claims per major section.
             When it exceeds the cap, compress the oldest third (drop
             section-level claims, keep only headings + slugs).
          3. **Defer all wiki writes to end-of-pass.** Maintain an
             in-memory ledger of concept/entity slugs encountered per
             chunk (don't create stub files mid-pass — you can't see
             whether a later chunk redefines or refines the term).
          4. After the last chunk: merge the ledger (de-dup slug variants
             like `multi-head-attention` vs `multihead-attention`,
             prefer the canonical form), then create stubs / update
             pages / flag contradictions in a single atomic final pass.
          5. Produce **one** source summary from the merged running
             outline. Cite section granularity in the summary
             (`(§3.2)`, `(ch. 4)`) so future reads can locate claims.
        - **>200k tokens** (books, theses, long reports) — **stop before
          writing anything** and propose a chapter-split plan: one source
          summary per chapter or per logical unit, slug suffixed `-ch01`,
          `-ch02`, etc. Wait for my go-ahead before ingesting.
      - Standalone image without a companion document (markdown, PDF,
        DOCX, etc.): tell me and stop. Images alone are not a source.
      - Anything else (audio, video, archives, structured data like raw
        JSON/CSV/XML, URLs): tell me and stop. Propose how to handle it
        before ingesting — these formats aren't in the default INGEST
        scope even though `markitdown` can convert some of them.
   b. Create `wiki/sources/{slug}.md` from the right template:
      - `templates/paper-summary.md` for anything in `raw/papers/` (academic
        papers, book chapters, long-form PDFs with citations).
      - `templates/source-summary.md` for articles, transcripts, and
        everything else.
   c. Extract concepts and entities. For each:
      - **Does not exist** → create a stub from `templates/stub.md`. Every
        `[[wikilink]]` you write must resolve by end of ingest — no dangling
        links permitted.
      - **Exists as a stub** → append this source to its backlinks. Inline
        promotion (rewrite as a full page) happens in step 3's whole-vault
        sweep below, which sees the page's full post-ingest state. This
        avoids the previous failure mode where a stub crossed the threshold
        via *another* source's later citation and never got promoted,
        because no single ingest saw the new total.
      - **Exists as a full page** → append new information. Never rewrite
        prose that hasn't changed.
   d. Contradictions: when new information conflicts with an existing page,
      insert a `> [!warning] Contradiction` callout at the point of conflict
      citing both sources. Never silently reconcile. See the canonical rule
      below.
3. **Whole-vault post-pass** (runs once after all sources in step 2 are
   processed):
   a. Run `python3 scripts/build_meta.py` — this recomputes
      `source_count` and `inbound_refs` for every concept/entity into
      `wiki/.meta/backlinks.json` in one O(N) pass. Reconcile frontmatter
      with the sidecar for any concept/entity whose values drifted.
   b. **Do not auto-promote stubs.** List every stub that crosses either
      promotion threshold ("Page creation thresholds" below) as a
      *promotion candidate* in the report. The user runs `/wiki-promote`
      to act on them. Auto-promotion inside INGEST burned Claude tokens
      on pages that weren't yet load-bearing; gating makes cost visible.
      Promotion authority now lives only in `/wiki-promote`.
   c. Set `last_seen: YYYY-MM-DD` on every page this ingest substantively
      touched: the new source summary, every stub created, every full
      page appended. **Do not** bump `last_seen` on pages hit only by an
      outbound wikilink resolution — that's write amplification.
4. Run `python3 scripts/shard_index.py` to rebuild `wiki/index.md`, the
   per-type `_index.md` shards, and tag shards.
5. Append one entry per source to `wiki/log.md`.
6. Report: sources processed, pages created, pages updated, stubs created,
   **promotion candidates**, contradictions flagged.

### QUERY
Trigger: `/wiki-query <question>` or I ask a question in chat.

Retrieval ladder — **do not load `wiki/index.md` at any step**. At 2000
sources, `index.md` would cost ~40K tokens per query before reading a
single page; that is exactly what the thin-index design prevents:

1. **Grep.** `python3 scripts/wiki_search.py "<question>"` — returns up
   to 10 ranked paths. If ≥3 hits with score ≥1.0, use them as the
   candidate set. Otherwise step 2.
2. **Tag shards.** If the question has an obvious topical angle, read
   `wiki/indexes/by-tag/<tag>.md` (if present). Add its listed pages to
   the candidate set.
3. **Type catalog.** Read `wiki/<type>/_index.md` for the most
   plausible type (source for factual, concept for definitional, entity
   for "who is X").
4. **Fallback:** only if 1–3 all came up empty, read `wiki/index.md`
   and give up if that's also dry (print "No coverage.").

Then:
5. Read every candidate page (cap ~10).
6. Synthesize. Cite every non-obvious claim with a `[[wikilink]]` to a
   source summary or concept page. Never invent citations.
7. Save to `wiki/outputs/{question-slug}.md` using `templates/output.md`.
8. Set `last_seen: YYYY-MM-DD` on the new output page and every page
   *cited* in the answer (not every page read — only cited).
9. Run `python3 scripts/build_meta.py` and `python3 scripts/shard_index.py`
   so the new output shows up in the sidecar and outputs index.
10. Append to `wiki/log.md`.

### LINT
Trigger: `/wiki-lint` or weekly.

Tiered scan (staleness-gated — do not re-read what was read recently):
- **Tier 1 — frontmatter only.** Read first ~30 lines of every page.
  Check required fields, shape. Cheap; always runs.
- **Tier 2 — full body.** Only for pages that failed Tier 1, pages
  whose `last_seen` is >7 days old, or new pages since last lint.
- **Tier 3 — cross-page contradictions.** Dispatch a subagent that
  reads pairs of pages sharing ≥2 cited sources and returns a
  contradiction report. Parent session never sees the raw bodies.

Steps:
1. `bash scripts/check_invariants.sh` — run the invariant suite first.
   If it fails, fix the filesystem before continuing.
2. `python3 scripts/build_meta.py` — refresh the sidecar.
3. Tier-1 fixes (frontmatter only):
   - Missing/malformed frontmatter (add required fields, preserve
     existing values).
   - Missing `last_seen` (initialize to `date_modified`).
   - Drifted `source_count` / `inbound_refs` — rewrite from
     `wiki/.meta/backlinks.json`.
4. Tier-2 fixes (full body; only for pages that need it):
   - Broken wikilinks → create stubs from `templates/stub.md`.
   - Inconsistent filenames → rename + update refs.
   - Duplicate pages under different names → merge.
   - `.base` / `.canvas` validity.
5. List stubs that cross either promotion threshold as
   **promotion candidates** — LINT does **not** auto-promote. Only
   `/wiki-promote` has promotion authority.
6. Tier-3 subagent: cross-page contradiction sweep. Report as inline
   `> [!warning]` callouts + a list in the report.
7. Report (do not auto-fix):
   - Orphan pages (no inbound wikilinks).
   - Stale content: `last_seen` older than 180 days (not `published`).
   - **Synthesis candidates:** clusters of ≥3 outputs sharing ≥2 cited
     pages.
8. Set `last_seen` on every page LINT substantively edited.
9. Re-run `python3 scripts/build_meta.py` and `python3 scripts/shard_index.py`.
10. Re-run `bash scripts/check_invariants.sh` — must pass after a clean
    lint. If it doesn't, report as a critical failure.
11. Write `wiki/outputs/lint-report-{today}.md`. Append to log.

### COMPILE
Trigger: `/wiki-compile` when you suspect index drift.
Steps:
1. `python3 scripts/build_meta.py` — rebuilds the sidecar
   (`wiki/.meta/backlinks.json`, `sources.json`, `search-index.tsv`).
2. `python3 scripts/shard_index.py` — rebuilds `wiki/index.md`,
   per-type `_index.md` (auto-sharding if needed), and
   `wiki/indexes/by-tag/*.md`.
3. Reconcile frontmatter `source_count` / `inbound_refs` with the fresh
   sidecar. These two fields are the only body-adjacent frontmatter
   COMPILE is allowed to touch; page body is never modified.
4. Append a `compile` entry to `wiki/log.md` with counts.

### PROMOTE
Trigger: `/wiki-promote [slug | --all | --list]`.

Explicit, user-gated graduation of stubs to full pages. Never fires
automatically. See `.claude/commands/wiki-promote.md` for the full
protocol and quality bar.

## Page creation thresholds
A stub graduates to a full page when **either** axis is met:

- **Source axis:** `source_count ≥ 2` — the subject appears in ≥2 raw
  sources (i.e. is wikilinked from ≥2 distinct `wiki/sources/*.md` pages).
- **Reference axis:** `inbound_refs ≥ 5` — the subject is wikilinked from
  ≥5 distinct pages anywhere in `wiki/`. Catches concepts that are
  load-bearing inside one deep source or structurally central to the vault
  even when only one source has covered them so far.

Both axes are evaluated by the Ingest post-pass (step 3) and by Lint.

- Stub: subject appears in exactly 1 raw source but is referenced by a
  `[[wikilink]]`. Must include frontmatter + one-sentence definition + link
  back to the source that mentioned it.
- Never: leave a `[[wikilink]]` pointing to nothing.

## Contradictions (canonical)
Ingest flags contradictions it encounters **while touching pages in this
ingest** (inline `> [!warning] Contradiction` callout, cites both sides).
Lint does the **cross-page sweep** across the entire wiki and issues fresh
callouts for any it finds plus a list in the lint report. Neither operation
resolves a contradiction silently — resolution is my call.

## Context engineering (canonical)

The 1M context window is double-edged: one session can do a whole batch,
but attention degrades as context grows (context rot is observable around
300–400K tokens depending on task). These rules keep operations cheap
at any vault size. Full detail in
`.claude/skills/context-engineering/SKILL.md`.

- **Subagents are the INGEST cost lever.** Batches of 3+ sources
  dispatch one subagent per source; the parent only sees structured
  reports. Sources >50K tokens always go to their own subagent.
- **Grep-first retrieval.** `wiki_search.py` before any index read.
  `wiki/index.md` is consulted last, not first. Candidate pages are
  read (at most ~10); the rest of the vault stays on disk.
- **Staleness gates.** LINT reads frontmatter first, full bodies only
  where signals demand. Pages with recent `last_seen` skip re-read.
- **Prompt-cache friendliness.** Read `CLAUDE.md` → skill references →
  stable catalogs → mutable candidate pages → output write. The stable
  prefix benefits from Anthropic's 5-min prompt cache TTL on back-to-back
  operations.
- **Do not autocompact inside an operation.** Mid-ingest compaction
  can summarise away the wikilink ledger and leave dangling links.
  Prefer `/clear` with a handoff brief between operations over
  mid-operation `/compact`.
- **`/rewind` beats "try again."** If an ingest picks the wrong
  template or a query reads the wrong candidate set, rewind to just
  before the mistake rather than trying to correct in-conversation.

## Quality bar
- Source summaries (articles, transcripts): 200–500 words, synthesis not
  transcription.
- Paper summaries: 300–800 words; **must** include bibliographic frontmatter
  and inline page citations on quotes and specific claims: `(p. 7)` or
  `(pp. 7–9)`.
- Concept / entity pages: 300–1200 words once promoted; 1 line for stubs.
- Every claim traceable to a specific source page.
- Figures: describe in prose by default. Extract to
  `wiki/attachments/images/` only when I explicitly ask, or when the figure's
  structure genuinely can't be captured in words (e.g. novel diagrams).
- Equations: use Obsidian's native MathJax. Inline math: `$...$`. Display
  (block) math: `$$...$$` on its own lines. **Do NOT wrap math in backticks** —
  backticks render as inline code (monospace), not typeset math. Use math
  mode for symbols and expressions ($d_{\text{model}}$, $QK^\top$,
  $\sqrt{d_k}$, $O(n)$, $10^{89}$, $E = mc^2$, etc.), not for programming
  constants or CLI strings (those stay in backticks). When an equation is
  nontrivial, follow it with a plain-language summary of what it's saying.
- When sources conflict, prefer recency, but flag with ⚠️ and preserve both.
- Avoid prose that could apply to any topic. If a paragraph would read the
  same for a different concept, rewrite it.

## Things you must not do
- Modify **existing** files in `raw/`. Creating new files in `raw/` via `/wiki-add` is allowed; editing or deleting content the user placed there is not.
- Edit `CLAUDE.md` itself, **except** during `/wiki-init` (first-run topic setup) or `/wiki-new-template` (teaching INGEST about a new source type). Never silently edit it during regular operation.
- Rewrite `wiki/log.md`.
- Use Dataview, Canvas, or Bases as **load-bearing** structure. The
  authoritative index (`wiki/index.md`), changelog (`wiki/log.md`), every
