# wiki/.meta/ — sidecar cache

Do not hand-edit. Regenerated by `scripts/build-meta.sh` (invoked by
`/wiki-ingest`, `/wiki-compile`, `/wiki-lint`).

## Files

- `backlinks.json` — inbound-link graph. Keys are page slugs; values
  include `inbound_sources` (from `wiki/sources/*.md`), `inbound_all`
  (every page), `source_count`, `inbound_refs`. Consumed by all
  operations that need to know "how many times is X cited."
- `sources.json` — flat list of sources with `slug`, `path`,
  `source_path`, `title`, `summary`, `tags`, `date_modified`,
  `last_seen`. Used to detect raw files awaiting ingest.
- `search-index.tsv` — tab-separated: `path \t title \t tags \t
  concepts \t summary`. One line per page across sources / concepts /
  entities / syntheses / outputs. The grep substrate for
  `scripts/wiki-search.sh`.
- `last_seen.json` — batched `last_seen` updates (so individual
  frontmatter writes stay quiet in git history). Flushed into page
  frontmatter by `/wiki-lint`.

## Why a sidecar, not frontmatter

`source_count` and `inbound_refs` live in concept / entity
frontmatter too — that is the canonical human-readable value. The
sidecar is the fast-read cache used by tooling to avoid `O(N²)`
rescans during INGEST and LINT. Frontmatter is authoritative; the
sidecar is derived. If they disagree, LINT rewrites frontmatter from
the sidecar and re-derives the sidecar from filesystem scan.

## When regenerated

- `/wiki-ingest` — incremental update for the ingested source's
  outbound links; full rebuild only in the post-pass.
- `/wiki-compile` — full rebuild of all three files from filesystem.
- `/wiki-lint` — full rebuild + consistency check between sidecar
  and frontmatter.
- `scripts/build-meta.sh` — one-shot full rebuild. Safe to run any
  time; idempotent.
