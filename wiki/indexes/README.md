# wiki/indexes/ — generated shard catalogs

Do not hand-edit. Regenerated by `scripts/shard-index.sh` (invoked by
`/wiki-compile` and `/wiki-lint`).

## Shape

- `wiki/index.md` — top-level nav. Counts + links to per-type
  `_index.md` files + links to shards here. Stays under **1K tokens**
  regardless of vault size. This is what `/wiki-query` reads first.
- `wiki/sources/_index.md`, `wiki/concepts/_index.md`,
  `wiki/entities/_index.md`, `wiki/syntheses/_index.md`,
  `wiki/outputs/_index.md` — per-type catalogs. Each stays under
  ~10K tokens; once it exceeds that, `shard-index.sh` splits it into
  alphabetical sub-shards (`_index-a-h.md`, `_index-i-p.md`,
  `_index-q-z.md`) and rewrites the parent to link to the shards.
- `wiki/indexes/by-tag/{tag}.md` — one file per tag with non-trivial
  membership. Used when `/wiki-query` has a clear topical angle
  (e.g. `by-tag/transformers.md`).
- `wiki/indexes/by-domain/{domain}.md` — grouped by top-level tag
  domain (e.g. `by-domain/ml.md` covers `ml-*` and adjacent tags).

## Why

The original design loaded `wiki/index.md` into context on every
QUERY. At 500 sources it is 10K tokens per query. At 2000 sources
it dominates the prompt. Sharded indexes let `/wiki-query` read only
the slice it needs.

## Regeneration cost

Full regeneration is O(N) in page count. On a 5000-page vault the
script finishes in under a second; it shells out to `rg` + `awk` and
does not invoke Claude.
